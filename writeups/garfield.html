Executive Summary

Garfield is a Hard-difficulty Windows Active Directory machine that demonstrates a sophisticated attack chain exploiting multiple AD misconfigurations and Kerberos weaknesses. The attack progresses from ACL abuse → logon script execution → lateral movement → RODC compromise → RBCD exploitation → Kerberos Golden Ticket forgery → KeyList attack → full domain administrative compromise.

Encoding Details:

  • Base64 wrapped to bypass execution policies

Vulnerabilities & CVEs

1. Active Directory ACL Misconfiguration — Type: Access Control Weakness · CVSS: High (7.2)

Users granted WRITE permission on other user objects (specifically the scriptPath attribute), enabling privilege escalation from a low-privilege user to arbitrary code execution.

Remediation: Audit AD ACLs regularly (e.g. BloodHound), apply least privilege to ACL assignments, monitor and alert on ACL modifications, use AD audit policies, track scriptPath changes via SIEM.

2. Logon Script Code Execution (scriptPath) — Type: Execution via Scheduled Script · CVSS: High (8.1)

The AD scriptPath attribute executes scripts automatically on user login with no content validation or sandboxing, enabling RCE as the authenticating user.

Remediation: Disable scriptPath if unused, restrict via GPO, monitor SYSVOL for unauthorized script changes, use AppLocker/Device Guard, audit logon scripts via GPO.

3. Machine Account Privilege (SeMachineAccountPrivilege) — Type: Excessive Privilege Assignment · CVSS: High (7.5)

SeMachineAccountPrivilege allows creating up to 10 computer objects by default, enabling RBCD attacks and lateral privilege escalation.

Remediation: Restrict the privilege to administrative accounts only, monitor computer object creation, lower quotas via Group Policy, audit "Add Workstations to Domain" operations.

4. RBCD Abuse — Type: Kerberos Protocol Weakness · CVSS: High (8.1)

RBCD allows flexible delegation without SeEnableDelegationPrivilege — write permission on a target object plus a controlled delegating account is enough for privilege escalation via S4U2Proxy.

Remediation: Restrict write permission on computer objects, monitor PrincipalsAllowedToDelegateToAccount changes, implement Kerberos Armoring, use the Protected Users group, audit S4U2Proxy operations.

5. RODC Golden Ticket + KeyList Attack — Type: Kerberos Cryptographic Weakness · CVSS: Critical (9.8)

An RODC maintains a separate krbtgt key, and the DC trusts RODC-issued tickets through the KeyList mechanism without validating that the RODC actually issued the ticket — a forged RODC ticket plus a KeyList request yields a real DC ticket.

Remediation: Secure RODC physical/logical access, monitor for credential extraction attempts, restrict msDS-RevealOnDemandGroup membership, implement PKINIT, monitor /keyList TGS requests, rotate krbtgt keys regularly, monitor LSA access.

6. Insufficient Clock Skew Tolerance — Type: Configuration Weakness · CVSS: Medium (5.3)

A multi-hour clock skew from the DC forced an NTP sync before Kerberos operations would succeed at all.

Remediation: Implement NTP synchronization across infrastructure, monitor for skew, use centralized Windows Time Service, alert on skew >5 minutes, secure NTP infrastructure.

7. Cleartext Credential Storage in Replication — Type: Cryptographic Weakness · CVSS: High (8.1)

The RODC's local password cache is extractable via Mimikatz once SYSTEM is achieved, effectively exposing cached credentials in plaintext-equivalent form.

Remediation: Use HSM-backed RODC deployments, restrict SYSTEM-level access, keep RODCs off the internet-facing perimeter, enable BitLocker and Credential Guard, monitor RODC access logs.

8. SID History and PAC Validation Bypass — Type: Kerberos PAC Weakness · CVSS: Critical (9.8)

The Golden Ticket's PAC contains group SID and rights information with no cryptographic binding checked by the RODC ticket path, letting arbitrary group membership and privilege be forged.

Remediation: Enable KDC PAC validation, PKINIT, and FAST; monitor for tickets with unusual PAC content; track TGT issuance anomalies via AD audit policies.

Defensive Recommendations

Short-Term (Immediate)

  • Audit Active Directory ACLs with BloodHound — identify WRITE permissions on critical objects and review scriptPath attributes on all users
  • Review RODC configuration (msDS-RevealOnDemandGroup / msDS-NeverRevealGroup)
  • Monitor for scriptPath modifications, computer object creation by non-admins, S4U2Proxy requests, and /keyList TGS-REQ operations
  • Rotate krbtgt keys twice, 10+ hours apart, to fully invalidate outstanding tickets

Medium-Term (1–3 months)

  • Implement Kerberos hardening: PKINIT, FAST, Kerberos Armoring
  • Audit and tighten all RBCD/delegation configurations
  • Deploy Kerberos-aware SIEM rules and monitor for Mimikatz-style LSA operations
  • Restrict SeMachineAccountPrivilege and adopt Just-In-Time admin access via the Protected Users group

Long-Term (3–12 months)

  • Modernize the AD estate (Windows Server 2022+, managed directory services)
  • Move toward a Zero Trust architecture with continuous authentication and network segmentation
  • Build Kerberos attack playbooks and train the SOC on Golden Ticket / KeyList indicators

Conclusion

Garfield demonstrates a sophisticated chain of vulnerabilities in Active Directory, Kerberos, and delegated authentication mechanisms. The attack successfully escalates from a low-privilege user to domain administrator by chaining together ACL misconfigurations (scriptPath write), logon script execution, privilege abuse (SeMachineAccountPrivilege), delegation flaws (RBCD), Kerberos weaknesses (Golden Ticket forgery), and replication cache abuse (KeyList attack).

The vulnerabilities are not isolated; each enables the next attack in the chain. Defending against such attacks requires defense-in-depth across multiple layers: ACL hardening, Kerberos armoring, privilege restriction, and comprehensive monitoring.

Key Takeaway: A single compromised system (RODC) combined with replication-based trust mechanisms can compromise the entire forest. Organizations must treat RODC security with the same rigor as primary domain controllers.
