Vulnerability Exploitation Framework

Automated Pentesting Toolkit — Python, Bash, YAML
sqlmap Dalfox Nuclei ParamSpider LFImap
Parameter Discovery → Auto-Routing → Module Execution → Reporting
↗ View source on GitHub
01

Problem

Manual pentesting workflows repeat the same sequence for every target — discover parameters, decide which vulnerability classes are plausible, run the matching tool, then write up findings. Doing this by hand across a large parameter surface doesn't scale.

Why this matteredThe goal wasn't to replace manual testing judgment, but to remove the repetitive triage work so more time goes to the parts that actually need a human — verifying findings and chaining them.
02

Design

The framework centers on an execution engine that coordinates scanning and exploitation modules, with ParamSpider handling automated parameter discovery across a target's attack surface.

Why this matteredCentralizing orchestration in one engine (rather than a loose collection of shell scripts) made it possible to add authenticated scanning and new modules later without rewriting the discovery layer.
03

Implementation

Auto-routing logic maps each discovered parameter to the appropriate vulnerability module — SQLi, XSS, LFI, SSRF, or IDOR — and dispatches it to the matching integrated tool: sqlmap for SQLi, Dalfox for XSS, LFImap for LFI, and Nuclei for broader template-based scanning. Custom scanners handle SSRF and IDOR, which don't have a strong off-the-shelf tool with the same detection tailoring. Authenticated scanning is supported through session handling and login automation so restricted areas of an app aren't skipped.

Why this matteredSSRF and IDOR needed custom detection logic because generic scanners produce too many false positives on these classes — a tailored check on response behavior gave more reliable signal than a one-size-fits-all payload list.
04

Result

The framework outputs structured HTML reports with findings and supporting evidence, turning a raw scan run into something that reads like an actual engagement deliverable rather than a wall of tool logs.

Why this matteredReporting was treated as a first-class part of the pipeline, not an afterthought — a scan result nobody can read quickly is a scan result nobody acts on.
