Project · Web Security

Secure Authentication System

Sentinel Hub — Flask 2FA & Role-Based Access Control
Flask bcrypt TOTP / 2FA RBAC SQLite
Password Hashing → TOTP 2FA → Session Tokens → Role-Based Routing
↗ View source on GitHub
01

Problem

Built for a Data Integrity and Authentication course assignment: implement a complete, real-world secure web authentication system rather than a bare login form — covering password storage, a second authentication factor, session protection, and enforced access control across distinct user roles.

Why this matteredAuthentication is one of the few areas of a web app where a shortcut isn't just a bug — plaintext passwords, missing 2FA, or unenforced role checks are each, on their own, a full account-takeover or privilege-escalation path.
02

Design

The system is a Flask app backed by SQLite, with registration collecting name, email, password, and one of exactly three roles — Admin, Manager, and User — each mapped to a distinct set of accessible routes rather than a single flat permission flag.

Why this matteredModeling roles as first-class, route-scoped permissions from the start (rather than bolting on an is_admin boolean later) keeps the access-control logic centralized and easier to audit.
03

Implementation — Hashing, 2FA, and Sessions

Passwords are hashed with flask-bcrypt and never stored in plaintext. On top of that, pyotp generates a per-user TOTP secret compatible with Google Authenticator, rendered as a scannable QR code via PyQRCode during enrollment. After a successful password check and TOTP verification, a secure session token is issued to track the authenticated user for all subsequent requests.

Why this matteredLayering TOTP on top of hashed passwords means a leaked or guessed password alone isn't enough to authenticate — the attacker also needs the physical/app-based second factor, which is the entire point of 2FA.
04

Implementation — Enforcing Role-Based Access

Each protected route checks the authenticated session's role before rendering: Admins get full user management (view, update roles, delete), Managers get read-only access to reports, and standard Users are limited to their own dashboard and profile. Unauthorized access attempts — a User requesting an admin-only route, for example — are strictly blocked with a custom 403/404 response rather than silently redirecting.

Why this matteredAccess control has to be enforced server-side on every request, not just hidden in the UI — a User who guesses or bookmarks an admin URL should hit an explicit denial, not a page that happens to render blank.
05

Result

The finished system covers registration, hashed-password login, TOTP-based 2FA enrollment and verification, session-token protected routing, and strictly enforced RBAC across three roles, with a clean dark-mode UI on top of a modular Python/Flask backend.

Why this matteredTreating this as a small, real authentication system — not just an assignment checklist — meant every piece (hashing, 2FA, sessions, RBAC) had to actually interlock correctly rather than working in isolation.
